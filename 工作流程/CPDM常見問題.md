## 基本概念

### Q1: CPDM 與傳統開發方法有什麼不同?

**A**: 主要差異在於:

| 面向 | 傳統方法 | CPDM |
|------|---------|------|
| 需求確認 | 多次會議 + 文字描述 | 視覺化雛型驗證 |
| 工具使用 | 單一工具 | 三大工具適配角色 |
| 進度追蹤 | 人工報告 | Antigravity 自動記錄 |
| 知識管理 | 分散文檔 | 結構化知識庫 |
| 返工率 | 20-30% | < 5% |

**核心優勢**: 降低溝通成本、快速驗證、進度透明

---

### Q2: CPDM 適合什麼類型的專案?

**A**: 最適合:
- ✅ 企業內部管理系統
- ✅ 資料輸入與報表系統
- ✅ 多部門協作專案
- ✅ 需要快速原型驗證的專案

**不太適合**:
- ❌ 單人小型專案 (工具成本 > 效益)
- ❌ 純演算法開發 (不涉及 UI)
- ❌ 已有明確規格的專案 (雛型驗證價值低)

---

### Q3: 團隊規模多大才適合使用 CPDM?

**A**: 建議團隊規模:
- **最小**: 3 人 (1 主管 + 1 開發者 + 1 工程師)
- **理想**: 5-10 人 (1-2 主管 + 1 開發者 + 3-7 工程師)
- **最大**: 15 人 (需要分組管理)

**注意**: 人數太少工具成本高,人數太多溝通複雜

## 工具使用

### Q4: 如何將 Canvas 設計傳送到 AI Studio?

**A**: 三種方法:

**方法 1: 結構化文字** (推薦)
```markdown
1. 在 Canvas 完成設計
2. 按照模板整理成結構化文字
3. 複製貼上到 AI Studio
```

**方法 2: 分享連結**
```markdown
1. Canvas 產生分享連結
2. 在 AI Studio 中引用連結
3. AI Studio 解析設計
```

**方法 3: 截圖 + 描述**
```markdown
1. Canvas 截圖
2. 加上文字描述
3. 一起貼到 AI Studio
```

**最佳實踐**: 使用方法 1,確保資訊完整

---

### Q5: AI Studio 產生的雛型系統可以直接開發嗎?

**A**: ❌ 不可以直接開發

**正確流程**:
1. AI Studio 產生雛型 → 用於**驗證需求**
2. 驗證通過後 → 開發者在 Antigravity **重構**
3. 重構產出架構文檔 → 工程師才開始開發

**原因**: 雛型是原型,不是生產代碼,需要專業的架構設計

---

### Q6: Antigravity 對話記錄如何用於進度追蹤?

**A**: 三個層面:

**1. 對話次數** = 工作量指標
```
工程師 A: 15 次對話 = 約 12 小時工作
```

**2. 對話主題** = 進度指標
```
對話 1-5: 畫面 A 開發
對話 6-10: 畫面 B 開發
對話 11-15: 整合測試
→ 進度 100%
```

**3. 重複主題** = 卡關指標
```
對話 12-20: 持續討論 API 整合
→ 卡關,需要介入
```

## 流程執行

### Q7: 需求定義階段一定要 1 週嗎?可以更快嗎?

**A**: 可以調整,但不建議太快

**時間分配**:
- **最快**: 3 天 (簡單系統 + 經驗豐富團隊)
- **標準**: 5-7 天 (一般系統)
- **複雜**: 10-14 天 (複雜系統 + 多部門)

**不建議太快的原因**:
- 雛型需要時間驗證
- 前端人員需要與業務確認
- 開發者需要充分思考架構

---

### Q8: 雛型驗證沒有達到 80% 滿意度怎麼辦?

**A**: 分析原因並調整

**常見原因**:
1. **需求理解偏差** → 與前端人員澄清 → 更新雛型
2. **功能遺漏** → 補充功能 → 重新驗證
3. **UI/UX 問題** → 調整設計 → 重新驗證
4. **效能疑慮** → 說明技術方案 → 建立信心

**重要**: 不要強行通過,返工成本更高

---

### Q9: 工程師在開發時可以直接改需求嗎?

**A**: ❌ 不可以直接改

**正確流程**:
```
發現問題 → 回報開發者 → 開發者評估
→ 需要改: 與主管討論 → 更新雛型 → 版本控制
→ 不需改: 說明理由 → 繼續開發
```

**原因**: 避免需求失控,保持系統一致性

## 問題處理

### Q10: 工程師卡關超過 4 小時怎麼辦?

**A**: 立即啟動支援機制

**步驟**:
1. **開發者介入** (30分鐘內)
   - 查看 Antigravity 對話記錄
   - 了解問題癥結
   
2. **技術討論** (1小時)
   - 開發者 + 卡關工程師
   - 分析問題 + 提供方案
   
3. **解決方案** (視情況)
   - 技術指導 → 工程師繼續
   - 重新分配 → 換人處理
   - 調整需求 → 與主管討論

4. **記錄知識** (30分鐘)
   - 問題 + 解決方案
   - 記錄到知識庫
   - 避免重複發生

---

### Q11: 發現技術無法實現雛型需求怎麼辦?

**A**: 及早發現,及早調整

**發現時機**:
- **最佳**: 步驟 3 架構設計時 (POC 驗證)
- **可接受**: 步驟 4 開發初期
- **最差**: 步驟 4 開發後期 (返工成本高)

**處理流程**:
```
開發者發現 → 準備替代方案 → 與主管討論
→ 調整需求 OR 調整技術 → 更新文檔
→ 重新評估時程 → 繼續開發
```

**預防**: 架構設計階段做充分的技術驗證

---

### Q12: 需求變更頻繁怎麼辦?

**A**: 版本控制 + 變更管理

**控制機制**:
1. **變更評估**
   - 影響範圍
   - 工時影響
   - 優先級

2. **版本規劃**
   - v1.0: 核心需求 (不變)
   - v1.1: 次要需求 (可調整)
   - v2.0: 新增需求 (下期)

3. **變更追蹤**
   - 變更追蹤表
   - 記錄原因
   - 影響分析

**原則**: v1.0 凍結後,新需求納入 v1.1 或 v2.0

## 團隊協作

### Q13: 前端人員不會用 Canvas 怎麼辦?

**A**: 提供培訓 + 簡化流程

**培訓計畫** (2小時):
1. Canvas 基本操作 (30分鐘)
2. 使用模板設計 (30分鐘)
3. 實際案例練習 (60分鐘)

**簡化方案**:
- 提供標準模板
- 填寫結構化表格
- 由開發者協助轉換

**注意**: Canvas 不是必須,重點是結構化的需求描述

---

### Q14: 主管沒時間整合雛型怎麼辦?

**A**: 調整角色 OR 簡化流程

**方案 1**: 開發者代理
```
開發者收集前端設計 → 在 AI Studio 整合
→ 產出雛型 → 主管驗證 → 分享前端
```

**方案 2**: 簡化驗證
```
前端設計 → 開發者直接重構
→ 產出架構 → 主管審查 → 開始開發
```

**注意**: 方案 2 風險較高,適合需求明確的專案

---

### Q15: 工程師之間如何協作?

**A**: 統一架構 + 定期同步

**協作機制**:
1. **統一架構文檔**
   - 所有人遵循同一份文檔
   - API 規格明確
   - 資料結構統一

2. **每日站立會議**
   - 15 分鐘快速同步
   - 識別依賴關係
   - 解決阻礙

3. **程式碼審查**
   - 完成子項目後審查
   - 確保品質一致
   - 知識分享

4. **Antigravity 記錄**
   - 技術決策透明
   - 問題解決可複用

## 品質管理

### Q16: 如何確保程式碼品質?

**A**: 多層次品質保證

**檢查點**:
1. **開發中**: 工程師自我檢查
2. **完成時**: 程式碼審查 (開發者 + 同儕)
3. **整合時**: 整合測試 (開發者 + QA)

**品質標準**:
- 功能符合需求
- 測試覆蓋率 ≥ 80%
- 程式碼可讀性良好
- 無明顯效能問題
- 文檔完整

**不通過**: 返回修改,重新審查

---

### Q17: 測試覆蓋率 80% 太高,可以降低嗎?

**A**: 可以調整,但不建議太低

**建議標準**:
- **核心功能**: 90% (資料處理、計算邏輯)
- **一般功能**: 80% (CRUD、查詢)
- **UI 元件**: 60% (視覺元件)

**最低底線**: 整體 ≥ 70%

**注意**: 測試覆蓋率不是唯一指標,測試品質更重要

---

### Q18: 如何評估專案成功?

**A**: 多維度指標

**專案級指標**:
- ✅ 準時交付: ≥ 90%
- ✅ 用戶滿意度: ≥ 85%
- ✅ 返工率: ≤ 10%
- ✅ Bug 率: ≤ 5%

**團隊級指標**:
- ✅ 卡關時間: ≤ 4h
- ✅ 知識累積: ≥ 10 篇
- ✅ 對話效率: ≥ 0.8h/對話

**業務級指標**:
- ✅ 業務目標達成
- ✅ ROI 正向
- ✅ 使用者採用率

## 進階問題

### Q19: CPDM 可以用於敏捷開發嗎?

**A**: ✅ 可以,而且很適合

**整合方式**:
- **Sprint 規劃**: 使用雛型驗證需求
- **每日站立**: CPDM 已包含
- **Sprint 回顧**: 整合到持續改進機制
- **增量交付**: 版本控制機制支援

**優勢**: CPDM 的雛型驗證加速敏捷的需求確認

---

### Q20: 如何持續改進 CPDM 流程?

**A**: 每個專案後回顧

**改進流程**:
```
收集數據 → 回顧會議 → 識別改進點
→ 制定行動計畫 → 更新 CPDM 文檔
→ 下個專案應用 → 驗證效果
```

**改進來源**:
- 專案指標數據
- 團隊回饋
- 問題記錄
- 最佳實踐

**記錄**: 所有改進記錄到知識庫
